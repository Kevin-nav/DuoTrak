### Overview of DuoTrak Backend for Onboarding

Your FastAPI backend will serve as the central API layer between your Next.js frontend and your Supabase (PostgreSQL, Auth, Storage) services. It will handle business logic, data validation, and secure interactions with Supabase, preventing the frontend from directly accessing sensitive Supabase APIs or database tables.

**Core Services Involved (FastAPI's Responsibilities):**

1.  **User Authentication & Management:**
    * **Delegation to Supabase Auth:** You will largely delegate actual user registration, login (email/password), and potentially social logins (Google, etc.) to Supabase Auth. The frontend will interact directly with the Supabase client-side SDK for these actions.
    * **Backend Validation & User Profile Creation:** After a successful registration/login on the frontend (via Supabase Auth), the frontend will send the Supabase access token to your FastAPI backend. FastAPI will validate this token, extract the `user_id`, and then create a corresponding user entry in your `users` table within the PostgreSQL database. This `users` table will hold DuoTrak-specific user metadata (e.g., username, partner status, onboarding completion).
    * **User Session Management:** The access token obtained from Supabase Auth will be sent with subsequent requests from the frontend to your FastAPI backend. FastAPI will use middleware to validate this token for every protected endpoint, ensuring only authenticated users can access them.

2.  **Partner Invitation & Linking:**
    * This is a core business logic that FastAPI will fully manage. It involves generating unique invitation codes, sending emails, and linking users.

**Database Tables (Supabase PostgreSQL) for Onboarding:**

* **`users` table:**
    * `id` (UUID, primary key, linked to `auth.users.id`)
    * `email` (Text, unique, from Supabase Auth)
    * `username` (Text, optional, if you allow custom usernames)
    * `onboarding_complete` (Boolean, default: `FALSE`)
    * `created_at` (Timestamp, default: `now()`)
    * `updated_at` (Timestamp)
    * `partnership_status` (Enum: 'none', 'pending_sent', 'pending_received', 'linked') - Tracks the user's current partner state.
    * `current_partner_id` (UUID, nullable, foreign key to `users.id`) - Links to the partner's user ID.

* **`invitations` table:**
    * `id` (UUID, primary key)
    * `sender_user_id` (UUID, foreign key to `users.id`)
    * `recipient_email` (Text)
    * `invitation_code` (Text, unique, generated by backend)
    * `status` (Enum: 'pending', 'accepted', 'rejected', 'expired')
    * `created_at` (Timestamp, default: `now()`)
    * `expires_at` (Timestamp)
    * `accepted_at` (Timestamp, nullable)

* **`partnerships` table:** (Alternative/complementary to direct `current_partner_id` on `users`)
    * `id` (UUID, primary key)
    * `user1_id` (UUID, foreign key to `users.id`, ensures lower ID is first to prevent duplicates)
    * `user2_id` (UUID, foreign key to `users.id`)
    * `established_at` (Timestamp, default: `now()`)
    * `status` (Enum: 'active', 'dissolved') - Allows for future features like dissolving partnerships.
    * *(You could simplify and just manage `current_partner_id` on the `users` table initially, and add this `partnerships` table later if more complex relationship management is needed.)*

### Guessed API Endpoints (FastAPI) for Onboarding

Here are the potential API endpoints your FastAPI backend will expose, along with their purpose and interaction:

**1. User Registration / Profile Creation:**

* **Endpoint:** `POST /api/v1/auth/register`
* **Purpose:** To create the initial user profile in your `users` table after Supabase Auth registration.
* **Request Body:** `{"access_token": "supabase_jwt_token", "username": "JohnDoe"}`
* **Response:** `{"user_id": "...", "username": "...", "onboarding_complete": false}`
* **Interaction:** Frontend receives JWT from Supabase Auth after sign-up, then immediately sends it to this backend endpoint to create the corresponding profile in your custom `users` table.

**2. User Profile Retrieval:**

* **Endpoint:** `GET /api/v1/users/me`
* **Purpose:** To retrieve the current authenticated user's profile and onboarding status.
* **Headers:** `Authorization: Bearer <supabase_access_token>`
* **Response:** `{"user_id": "...", "username": "...", "onboarding_complete": true, "partnership_status": "linked", "current_partner_id": "..."}`
* **Interaction:** Frontend calls this on app load or after login to get the user's profile and determine which onboarding step to show.

**3. Update Onboarding Status:**

* **Endpoint:** `PUT /api/v1/users/me/onboarding-complete`
* **Purpose:** To mark the onboarding process as complete for the user.
* **Headers:** `Authorization: Bearer <supabase_access_token>`
* **Request Body:** `{"onboarding_complete": true}` (or infer from endpoint call)
* **Response:** `{"message": "Onboarding status updated."}`
* **Interaction:** Frontend calls this once the user has finished all onboarding steps.

**4. Send Partner Invitation:**

* **Endpoint:** `POST /api/v1/partnerships/invite`
* **Purpose:** To send an invitation email to a potential partner.
* **Headers:** `Authorization: Bearer <supabase_access_token>`
* **Request Body:** `{"recipient_email": "partner@example.com"}`
* **Response:** `{"message": "Invitation sent successfully.", "invitation_id": "..."}`
* **Interaction:** Frontend captures the partner's email and sends it to this endpoint. The backend handles code generation, saving to `invitations` table, and sending the email. Also updates sender's `partnership_status` to 'pending_sent'.

**5. Accept Partner Invitation:**

* **Endpoint:** `POST /api/v1/partnerships/accept-invite`
* **Purpose:** To accept an invitation using the unique code received via email.
* **Headers:** `Authorization: Bearer <supabase_access_token>` (for the accepting user)
* **Request Body:** `{"invitation_code": "unique_code_from_email"}`
* **Response:** `{"message": "Partnership established!", "partner_id": "..."}`
* **Interaction:** Frontend prompts the accepting user to enter the code. Backend validates the code, links the users in the `users` and/or `partnerships` table, updates `invitations` status to 'accepted', and updates both users' `partnership_status` to 'linked'.

**6. Reject Partner Invitation:**

* **Endpoint:** `POST /api/v1/partnerships/reject-invite`
* **Purpose:** To explicitly reject an invitation.
* **Headers:** `Authorization: Bearer <supabase_access_token>` (for the rejecting user)
* **Request Body:** `{"invitation_code": "unique_code"}`
* **Response:** `{"message": "Invitation rejected."}`
* **Interaction:** Frontend allows the user to reject an invite. Backend updates the `invitations` status to 'rejected' and clears the sender's 'pending_sent' status if applicable.

### How Frontend (Next.js) Will Interact with Backend (FastAPI)

1.  **Direct Supabase SDK for Initial Auth:** For `supabase.auth.signUp()` and `supabase.auth.signInWithPassword()`, the Next.js frontend will use the `@supabase/supabase-js` client library directly. This handles the actual communication with Supabase's Auth service, session management in the browser (local storage/cookies), and returning a JWT (`access_token`).

2.  **`fetch` or Axios for FastAPI API Calls:** For all interactions with your custom business logic (user profile creation, sending/accepting invitations), the Next.js frontend will make standard HTTP requests (POST, GET, PUT) to your FastAPI server using JavaScript's `fetch` API or a library like Axios.

3.  **Authentication Flow for FastAPI:**
    * After a user successfully authenticates with Supabase Auth (e.g., signs in), the Next.js frontend will retrieve the `session.access_token` from the Supabase client.
    * This `access_token` will then be included in the `Authorization` header of *every* request made to your FastAPI backend: `Authorization: Bearer <access_token>`.
    * Your FastAPI backend will have a **middleware** that intercepts these requests, extracts the token, and uses the Supabase Python SDK to *verify* the token with Supabase's Auth service. If valid, the `user_id` is extracted and made available to your API endpoints (e.g., via `request.state.user_id`). This ensures all your backend endpoints operate on behalf of an authenticated and identified user.

4.  **Error Handling:** Frontend will display user-friendly messages based on responses (and error codes) from FastAPI.

5.  **Data Fetching Strategy (Next.js):**
    * You'll likely use **React Query (TanStack Query)** or Next.js's built-in data fetching (e.g., `fetch` directly in Server Components/Actions) for managing data from your FastAPI endpoints, including caching, loading states, and error handling.

### Connecting FastAPI to Local Supabase Backend Locally

As discussed, this is straightforward now that you have the Supabase CLI and Docker:

1.  **Start Local Supabase:** In your `DuoTrak_v1.1/backend` directory, run `supabase start`. This spins up the Docker containers and provides you with the local `DB URL`, `API URL`, and `keys`.

2.  **Environment Variables:** In your FastAPI project (`DuoTrak_v1.1/backend`), typically in a `.env` file, configure your application to use these local credentials:

    ```ini
    # DOCKER_HOST=tcp://127.0.0.1:2375 # Might be needed on some Windows setups if issues persist, but usually not with Docker Desktop.
    # SUPABASE_URL and KEYS are for the Supabase Python client if you use it to interact with Auth/Storage.
    SUPABASE_URL=http://localhost:54321
    SUPABASE_ANON_KEY=eyJ...
    SUPABASE_SERVICE_ROLE_KEY=eyJ...
    JWT_SECRET=your_local_jwt_secret_from_supabase_start

    # This is for SQLAlchemy/Alembic direct database connection
    DATABASE_URL="postgresql://postgres:postgres@localhost:54322/postgres"

    # You might also need a sender email and API key for your email service (e.g., SendGrid, Mailgun)
    # for sending invitation emails.
    EMAIL_SERVICE_API_KEY=your_email_service_api_key
    SENDER_EMAIL=noreply@duotrak.com
    ```

3.  **Alembic for Migrations:**
    * You'll initialize Alembic in your `backend` project if you haven't already: `alembic init alembic_migrations`.
    * Configure `alembic.ini` to point to the `DATABASE_URL` for your local Supabase.
    * Generate migrations as you make changes to your SQLAlchemy models: `alembic revision --autogenerate -m "description_of_change"`.
    * Apply migrations: `alembic upgrade head`.

This setup allows you to run your FastAPI application, have it connect to the local Supabase containers, and simulate the full backend functionality without touching your remote hosted project during development.
